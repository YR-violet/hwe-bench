# verify (s10)

`verify/` is the s10 stage that checks and repairs candidate benchmark cases from tbgen. The input is an s09 JSONL with `tb_script` and `prepare_script` fields. The output is an s10 JSONL containing only cases that pass the standard fail-to-pass validation.

The stage has two entry points:

1. `run_batch.py` prepares per-case artifact directories and asks Codex to repair scripts when needed.
2. `run_case.py` runs one prepared case through the standard Docker verifier.

## Codex Profile

`run_batch.py` launches `codex exec -p normal` through the shared `codex_batch.py` helper. Define a `normal` profile in `$CODEX_HOME/config.toml` or `~/.codex/config.toml` before running verify. The subprocess closes stdin explicitly, so it does not inherit non-terminating input from the parent process.

## Batch Verify

`run_batch.py` reads the s09 JSONL, creates `artifacts/s10_verify/{org}__{repo}/pr-{N}/` for each selected PR, writes the case files Codex needs, renders the repo-specific verify prompt, and dispatches Codex. The verify prompt is not a fresh tbgen task: it asks Codex to keep the existing scripts when possible, make the smallest needed repair, and call `run_case.py` for real validation.

```bash
uv run python -m hwe_bench.harness.verify.run_batch \
  --input datasets/pipeline/<ORG>/<ORG>__<REPO>_s09_tbgen_output.jsonl \
  --org <ORG> \
  --repo <REPO> \
  --out-root artifacts/s10_verify \
  --num-workers 1
```

By default, `run_batch.py` selects the prompt from `verify/prompts/{org}__{repo}.md`. Use `--prompt-template` for a custom prompt, `--only 123,456` for a subset, and `--force` to rerun cases that already have `result.json` with `status == "success"`.

If a previous failed verify attempt left edited `tb_script.sh` or `prepare_script.sh` in the case directory, a normal rerun keeps those files so Codex can continue from them. Use `--force` when the input JSONL should overwrite those scripts.

## Artifact Directory

Each case directory contains:

| File or directory | Purpose |
|-------------------|---------|
| `case.json` | Full case record from the s09 input |
| `case.jsonl` | Single-record dataset generated by `run_case.py` for `docker_runner` |
| `pr_meta.json` | Metadata and paths rendered into the prompt |
| `fix.patch` | Ground-truth fix patch |
| `test.patch` | Original test patch, if present |
| `tb_script.sh` | Candidate or repaired runtime script |
| `prepare_script.sh` | Candidate or repaired prepare override |
| `prompt_rendered.md` | Exact prompt passed to Codex |
| `result.json` | Codex verify result with final scripts and failure details |
| `workdir/` | Docker build and run workspace used by `run_case.py` |
| `logs/` | Build and run logs |

`run_batch.py` copies final script content from `result.json` back to `tb_script.sh` and `prepare_script.sh` after Codex exits. `result.json` remains the authoritative output consumed by the merge step.

## Single-Case Verification

`run_case.py` is the standard verifier for one prepared case directory. It reads `case.json`, rebuilds the one-record `case.jsonl`, clears the previous instance workspace, and invokes `docker_runner` in instance mode.

```bash
uv run python -m hwe_bench.harness.verify.run_case \
  --case-dir artifacts/s10_verify/<ORG>__<REPO>/pr-<N>
```

Use `--force-build` when `prepare_script.sh` changed and the per-PR Docker image must be rebuilt:

```bash
uv run python -m hwe_bench.harness.verify.run_case \
  --case-dir artifacts/s10_verify/<ORG>__<REPO>/pr-<N> \
  --force-build
```

`run_case.py` prints a JSON summary and exits with `0` only when the generated `report.json` has `valid == true`. A nonzero exit means the case still fails validation; inspect `workdir/{org}/{repo}/images/pr-{N}/build_image.log`, `workdir/{org}/{repo}/instances/pr-{N}/test-patch-run.log`, `fix-patch-run.log`, and `report.json`.

## Merge Results

After verify finishes, merge successful outputs into the s10 JSONL:

```bash
uv run python -m hwe_bench.harness.tbgen.merge_results \
  --input datasets/pipeline/<ORG>/<ORG>__<REPO>_s09_tbgen_output.jsonl \
  --artifacts-root artifacts/s10_verify \
  --output datasets/pipeline/<ORG>/<ORG>__<REPO>_s10_verified.jsonl \
  --success-only
```

The same merge helper is used for tbgen and verify because both stages write final script content into per-case `result.json` files.

## Prompt Files

Repository-specific verify prompts live in `verify/prompts/`. They explain the standard harness semantics for each repo, including where the repository lives in the container, how `prepare_script.sh` is baked into the image, how `tb_script.sh` is mounted at runtime, and how the log parser reads `TEST: ... PASS|FAIL|SKIP` markers.

OpenTitan verify prompts require the real VCS / `dvsim` / UVM-style flow when applicable. Chisel prompts for XiangShan and rocket-chip prefer focused Mill/SBT or subsystem-level flows over full-system simulation when the bug can be reproduced honestly at a smaller scope.
