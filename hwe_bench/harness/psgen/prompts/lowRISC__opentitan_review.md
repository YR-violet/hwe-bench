You need to review the **problem_statement** of a lowRISC/opentitan bug (generated by the previous stage) and ensure it is suitable as a benchmark issue description for an AI agent.

OpenTitan is a complete open-source Root of Trust SoC platform containing multiple IP modules (AES, HMAC, KMAC, OTBN, SPI, I2C, UART, GPIO, Flash Controller, OTP Controller, Alert Handler, CSRNG, EDN, Entropy Source, Power/Reset/Clock Manager, etc.) as well as system-level infrastructure (TL-UL bus interconnect, Pinmux, Padctrl, etc.).

## Your input

The following files already exist under the current directory `{PR_DIR}`:

- `problem_statement.json`: the problem_statement generated by the previous stage (the object you are reviewing)
- `case.json`: the full case record, containing fields such as title/body/resolved_issues. **Read `resolved_issues` from here.**
- `fix.patch`: the ground-truth fix patch
- `result.json`: results from the tbgen/verify stages (contains tb_script)

You may also use `gh` to inspect PR/issue discussions to help your judgment.

## Review checklist

### 1. Semantic completeness

The problem_statement must cover four elements: the observed anomalous behavior, the expected correct behavior, the affected functional module, and the trigger condition. If any element is missing, add it.

Among these, **expected behavior** must be a decidable, concrete normative description. Check for vague wording (e.g., "should work correctly", "should handle this properly") — if present, replace with specific, verifiable behavioral descriptions.

For OpenTitan, the affected functional module should be concrete to the IP level or submodule level (e.g., "AES core's key scheduling", "SPI Device firmware mode RX FIFO"), not merely "opentitan".

### 2. Self-containedness

The problem_statement must remain independently understandable once detached from the original issue's context. Check the following:

- Are there undefined terms or acronyms? OpenTitan's common acronyms should be expanded: OTP (One-Time Programmable), CSRNG (Cryptographically Secure Random Number Generator), EDN (Entropy Distribution Network), OTBN (OpenTitan Big Number Accelerator), TL-UL (TileLink Uncached Lightweight), KMAC (Keccak Message Authentication Code), LFSR (Linear Feedback Shift Register), CDC (Clock Domain Crossing), etc.
- Does it assume the reader has seen previous discussion or knows OpenTitan project-internal conventions?
- Is it necessary to add a minimum amount of IP design spec background to make the description independently understandable?

After reading the problem_statement, an agent unfamiliar with the opentitan project should be able to clearly understand what the bug is and what the correct behavior is.

### 3. Clarity and unambiguity

Check for room for reasonable misinterpretation. Specifically:

- If an agent unfamiliar with opentitan reads this description, could it arrive at a plausible but wrong understanding?
- Do any keywords have multiple meanings? (e.g., "host" in SPI refers to the master device, but in the system context refers to the processor; "key" in AES refers to the encryption key, but in OTP refers to a partition key)
- Is the trigger condition specific enough, or could the agent fix a different bug in the same IP?

If there is ambiguity, revise the wording so that only one reasonable reading is possible.

### 4. Behavior-level alignment with the test

Read the tb_script in `result.json` to understand the behavior it actually verifies. Then check alignment in two directions:

**problem_statement → tb_script**: for the bug behavior described in problem_statement, does tb_script have a corresponding check? If problem_statement mentions an anomalous behavior that tb_script does not test, then that part of the description is superfluous — the agent could pass the test without fixing that behavior. Consider removing or softening that part.

**tb_script → problem_statement**: for the behavior tb_script verifies, does problem_statement have a corresponding description? If tb_script checks a specific boundary condition (e.g., a particular register configuration, particular operation timing, particular mode combination) but problem_statement only gives a vague module name, the agent might fix the correct module but miss the specific scenario — which is unfair. Add a description of that boundary condition (at the behavior level, not at the test-method level).

Note: alignment here is at the behavior level. Do not describe how tb_script is set up inside problem_statement — just make sure the bug-behavior description and the tb_script's verification scope are consistent.

### 5. Information granularity

Use the level of detail in the original issues under `case.json`'s `resolved_issues` as reference. Information already in the original issue can be kept verbatim. But anything the previous stage added itself should not exceed the original issue's granularity.

Comparison method: put the problem_statement alongside `resolved_issues[*].body` in `case.json` and line them up — distinguish content that comes from the original issue (can be kept) from content the previous stage added itself (needs granularity checks).

### 6. Leakage check

The following, if present in content **added by the previous stage** (as opposed to appearing in the original issue verbatim), counts as leakage:

- Fix approach ("should change X to Y", "the FSM should transition to StIdle")
- Specific file names or line numbers not in the original issue
- Excerpts of fix.patch / tb_script
- Simulation/build commands or test setup methods ("run fusesoc ...", "make -C ...", "write a test that ...")
- Test names, test seeds, UVM assertion names, log paths

### 7. Information completeness

If the PR is linked to multiple `resolved_issues`, check whether the problem_statement covers all semantically relevant issues, rather than describing only one of them.

## Handling

- Passes review: keep as-is, mark status as `approved`
- Needs changes: revise and write back, mark status as `revised`, explain the reason
- Serious problems requiring rewrite: rewrite and write back, mark status as `rewritten`, explain the reason

Must be written in English.
